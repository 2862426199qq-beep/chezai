# FMCW 毫米波雷达 Bringup 全流程复盘

> **硬件**：RK3588S（鲁班猫）← SPI → STM32 ← FMCW 雷达前端  
> **协议**：24-26GHz FMCW，64 IQ 点/帧，帧头 `AA 30 FE FF`，帧尾 `FE FE FE 55`  
> **驱动版本线**：V0.1（骨架）→ V3.1（环形缓冲 + 阻塞读）→ V3.2（当前，atomic + 专属 workqueue + poll 支持）

## Phase 0：骨架验证 —— "先让设备节点出现"

**目标**：确认 SPI 总线能挂上、`/dev/fmcw_radar` 能创建。

**做了什么**：

写了一个最小化驱动 `try/standard_spi.c`（模块名 `radar1.ko`），只做三件事：
1. `probe()` —— 分配 `fmcw_radar_dev`，注册 `miscdevice`
2. `open()` —— 把设备指针存到 `file->private_data`
3. `read()` —— **空实现**，直接返回 0

```c
/* try/standard_spi.c 的 read —— 啥也不干，只是为了让 open/read 不报错 */
static ssize_t fmcw_radar_read(struct file *filp, char __user *buf,
                                size_t count, loff_t *ppos) {
    return 0;  // 现在啥也不读，就打开设备
}
```

**启动流程**（`try/start.sh`）：
```
sudo rmmod fmcw_radar              ← 先卸载旧的（如果有）
sudo insmod radar1.ko              ← 加载骨架驱动
echo fmcw_radar > driver_override  ← 告诉 spi0.0："你的驱动是 fmcw_radar"
echo spi0.0 > bind                 ← 触发 probe()
sudo chmod 666 /dev/fmcw_radar     ← 普通用户也能读
```

**验证**：`ls /dev/fmcw_radar` 存在 → `cat /dev/fmcw_radar` 不报错（虽然读不到数据）→ **Phase 0 通过**。

```
验证清单：
  [✓] insmod 不报错
  [✓] dmesg 里看到 "probe OK, /dev/fmcw_radar created"
  [✓] open() + read() 不崩
  [✓] rmmod 能干净卸载
```

## Phase 1：数据打通 —— "读到东西了，但全是乱的"

**目标**：从 SPI 总线上真正读到雷达数据。

**驱动改动**：往骨架里加入了完整的 SPI 通信链路——

```
    ┌─────────────────────────────────────────────────────┐
    │                 驱动内部数据流水线                    │
    │                                                     │
    │   hrtimer (1ms)                                     │
    │      │                                              │
    │      ▼                                              │
    │   queue_work()  ──→  fmcw_spi_work_func()           │
    │                          │                          │
    │                    spi_sync() 读 264 字节            │
    │                          │                          │
    │                    ring_buf_write()                  │
    │                          │                          │
    │                     环形缓冲区                       │
    │                    (256帧 × 264B)                    │
    │                          │                          │
    │              用户 read() ← ring_buf_read()          │
    │              阻塞等待 wait_event                     │
    └─────────────────────────────────────────────────────┘
```

**关键设计决策**：为什么不用 kthread？
- hrtimer 回调在硬中断上下文，**不能调 `spi_sync()`**（会睡眠）
- 所以拆成两层：hrtimer 只做 `queue_work()`，SPI 读取在 workqueue 线程里做
- 相当于"闹钟叫你起床（hrtimer），起床后自己去干活（workqueue）"

**调试工具 radar_debug.c**（第一把刀）：
```
读 3 次，每次 264 字节，打印：
  ① 前 16 字节（找帧头 AA 30 FE FF）
  ② 后 8 字节（找帧尾）
  ③ 全零字节占比（判断是不是全是空数据）
  ④ 扫描 AA 30 FE FF 出现的偏移位置
```

**现象**：
```
第 1 次读取：264 字节
  前 16 字节: 00 00 AA 30 FE FF 12 34 ...    ← AA 在偏移 2！不在偏移 0！
  ★ 在偏移 2 处发现帧头 AA 30 FE FF ！

第 2 次读取：264 字节
  前 16 字节: 56 78 00 00 AA 30 FE FF ...    ← 这次帧头跑到偏移 4 了！
```

帧头**漂移不定**，说明每次 SPI 读 264 字节时，并没有和 STM32 的帧边界对齐。

**调试工具 radar_debug2.c**（第二把刀）——更大视野：
```
一次读 2048 字节，搜索所有帧头和帧尾：
  ★ 偏移 2   处发现帧头
  ★ 偏移 266 处发现帧头         ← 间距 264，对了！
  ★ 偏移 530 处发现帧头         ← 又是 264，稳定！
  ★ 帧尾 FE FE FE 55 也找到了
```

**结论**：数据本身是**好的** —— 帧头帧尾都在、间距正确、IQ 数据非全零。问题出在**每次 `spi_sync` 的 264 字节读取并不总是从帧头开始**。

**调试工具 radar_dump.c**（第三把刀）——全量 hex dump：
读两帧（528 字节），完整打印每个字节 + 偏移量。用来人眼确认帧结构：

```
  AA 30 FE FF xx xx xx xx ...（256 字节 IQ）... FE FE FE 55
  |←── 4 ──→|←──── 256 ────────────────────→|←── 4 ──→|
  帧头         64 个 IQ 点（各 4 字节）         帧尾
  总计 264 字节
```

**调试工具 radar_dump2.c**（第四把刀）——帧长兼容检查：
这个工具同时检查 264 和 268 两种帧长（STM32 固件历史上有过 268 字节帧），打印帧序号和扫描角度字段：
```
帧头在偏移 0:
  帧序号: 1234
  扫描角度: 450 (= 45.0°)
  帧尾(266-267): FE 55
  ⚠ 帧尾在偏移264！STM32还在用264字节旧帧！   ← 后来确认主链路就是 264
```

## Phase 2：SPI 时序排查 —— "1bit 集体偏移"之谜

### 2.1 现象

帧头 `AA 30 FE FF` 虽然能在数据流中找到，但有时候会出现一种诡异现象：

```
正常数据：               AA  30  FE  FF  ...
错位数据（右移 1bit）：   55  18  7F  7F  ...

对比（二进制）：
  AA = 10101010          →  右移 1bit  →  01010101 = 55 ✗
  30 = 00110000          →  右移 1bit  →  00011000 = 18 ✗
```

**整帧所有字节统一偏移 1 个 bit** —— 不是某几个字节错，是**全军覆没**。

### 2.2 原因定位

```
  ┌─────────────────────────────────────────────────────┐
  │             SPI 时序边沿匹配问题示意                  │
  │                                                     │
  │  SCLK:  ──┐  ┌──┐  ┌──┐  ┌──┐  ┌──                │
  │           └──┘  └──┘  └──┘  └──┘                   │
  │                                                     │
  │  MISO:  ──╳──────╳──────╳──────╳──────              │
  │          D7     D6     D5     D4                    │
  │                                                     │
  │  mode0     ↑采样点（上升沿）                          │
  │  采样：    此时 MISO 还在翻转！踩在不稳定区域！        │
  │                                                     │
  │  mode1       ↑采样点（下降沿）                        │
  │  采样：      此时 MISO 已经稳定了 ✓                   │
  └─────────────────────────────────────────────────────┘

  CPOL = 0, CPHA = 0 (mode0)：上升沿采样，但 STM32 在上升沿才翻转数据
  CPOL = 0, CPHA = 1 (mode1)：下降沿采样，此时数据已稳定半个周期
```

虽然 STM32 标称 `SPI mode 0`，但实际时序行为更接近 `mode1` 的预期。
这在嵌入式开发中**极其常见**：芯片手册说的和实际波形经常差半拍。

### 2.3 调试工具 radar_debug3.c（终极确认）

滑动窗口累积读取，最多读 20 次，每次打印累积偏移信息：

```
第  1 次读取: +264 字节, 累计 264 字节
第  2 次读取: +264 字节, 累计 525 字节     ← 保留了上次末尾 3 字节防跨界
第  3 次读取: +264 字节, 累计 786 字节

  ★★★ 在累计偏移 2 处发现帧头 AA 30 FE FF ！★★★
  完整帧前 16 字节: AA 30 FE FF 12 34 56 78 ...
  完整帧后 8 字节:  ... FE FE FE 55
  ✅ 帧尾 FE FE FE 55 正确！
  ✅ 下一帧帧头也在！帧间距 = 264 字节！
```

用 radar_debug3 分别在 `SPI_MODE_0` 和 `SPI_MODE_1` 下各跑一轮：

```
  ┌───────────────────────────────────────────────────┐
  │              A/B 对比结果                          │
  │                                                   │
  │  指标              mode0          mode1            │
  │  ──────────────────────────────────────────────    │
  │  帧头命中率        不稳定，需偏移   首次即命中       │
  │  帧间距            264 但有跳变     稳定 264        │
  │  帧尾 FE FE FE 55  偶尔错位        每帧正确         │
  │  1bit 偏移体感      有              无              │
  │                                                   │
  │  结论：mode1 全面优于 mode0 ──→ 驱动锁定 mode1     │
  └───────────────────────────────────────────────────┘
```

### 2.4 最终修复

驱动 `fmcw_radar.c` 的 `probe()` 里写死：
```c
spi->mode = SPI_MODE_1;    /* CPOL=0, CPHA=1 */
spi_setup(spi);
```

## Phase 3：稳定运行 & 上层对接

### 3.1 驱动迭代到 V3.2

从 Phase 0 的骨架到最终版，驱动经历了以下演进：

```
  V0.1（try/）          V3.1                   V3.2（当前）
  ─────────────────────────────────────────────────────────────
  空 read()             环形缓冲 + 阻塞读       + atomic_t count
  无 SPI 通信           hrtimer + 系统 wq       + 专属 workqueue
  全局 dev 指针         每次 kmalloc read_buf    + 预分配 read_buf
  无 poll 支持          ——                      + poll/select/epoll
  ——                   全局指针                 + container_of 多设备
```

**V3.2 关键改进**：
- `count` 从 `int` 改为 `atomic_t` —— 修复 spinlock 外的裸读竞争
- 专属 `alloc_workqueue("fmcw_spi", WQ_HIGHPRI | WQ_UNBOUND, 1)` —— SPI 采集不被其他内核任务拖慢
- `read_buf` 预分配 —— 避免 1ms 一次的 `kmalloc/kfree` 导致内存碎片
- `poll()` —— Qt 应用通过 `QSocketNotifier` 用 epoll 监听设备

### 3.2 应用层验证 radar_test.c (V3.1)

## 当前使用方式

### 每次开机启动

执行 `start_radar.sh`：
```bash
sudo rmmod fmcw_radar 2>/dev/null                          # ① 清理残留
cd /home/cat/chezai/driver/fmcw_radar && sudo insmod fmcw_radar.ko  # ② 加载模块
echo fmcw_radar | sudo tee /sys/bus/spi/devices/spi0.0/driver_override  # ③ 绑定 spi0.0
echo spi0.0 | sudo tee /sys/bus/spi/drivers/fmcw_radar/bind            # ④ 触发 probe
sudo chmod 666 /dev/fmcw_radar                             # ⑤ 开放权限
```

### 驱动卸载顺序（probe 的逆序，不能乱）

```
  ① atomic_set(&alive, 0)     宣布死亡 → 定时器/工作项看到后停止
  ② wake_up_interruptible()   踢醒沉睡的 read()，防止僵尸进程
  ③ hrtimer_cancel()          停定时器 → 不再有新的 queue_work
  ④ cancel_work_sync()        等当前 SPI 读取结束（_sync 会阻塞等）
  ⑤ destroy_workqueue()       销毁专属工作队列
  ⑥ misc_deregister()         删除 /dev/fmcw_radar 节点
```

## 数据帧格式（最终确认 264 字节）

```
  Byte 偏移     内容                    说明
  ──────────────────────────────────────────────────
  0-3           AA 30 FE FF             帧头（固定魔数）
  4-259         IQ[0]...IQ[63]          64 个 IQ 点
                每点 4 字节:             I(int16) + Q(int16)
  260-263       FE FE FE 55             帧尾（固定魔数）
  ──────────────────────────────────────────────────
  总计          264 字节
```

> ⚠️ `radar_dump2.c` 中有 268 字节帧的检查逻辑（260-261 帧序号，262-263 角度，266-267 帧尾），
> 这是 STM32 固件的历史版本产物。当前主链路已确认为 264 字节帧。

## 同步与并发机制

```
  ┌──────────────────────────────────────────────────────────┐
  │                    三把锁各管各的                          │
  │                                                          │
  │  spinlock (ring_lock)                                    │
  │    └── 保护环形缓冲区 head/tail/count                     │
  │    └── 在中断上下文和进程上下文之间同步                     │
  │    └── 持有时间极短（只做指针移动 + memcpy）                │
  │                                                          │
  │  mutex (read_lock)                                       │
  │    └── 保护 read() 不被多个用户进程并发调用                 │
  │    └── 持有时间较长（包含 ring_buf_read → copy_to_user）   │
  │                                                          │
  │  atomic_t (count, alive)                                 │
  │    └── count: 缓冲区字节数，wait_event 的判断条件           │
  │    └── alive: 驱动生命周期标志，所有异步路径检查             │
  └──────────────────────────────────────────────────────────┘

  为什么 ring_buf_read 后再 copy_to_user，中间用 read_buf 中转？
  ──────────────────────────────────────────────────────────
  ring_buf_read() 持有 spinlock → copy_to_user() 可能缺页睡眠
  持有 spinlock 时睡眠 = 死锁！
  所以：spinlock 下只做 memcpy 到内核临时缓冲区，释放锁后再 copy_to_user
```

## 示波器验证清单（5 分钟快速版）

当再次遇到"1bit 集体偏移"时，按以下步骤排查：

### 1. 接线
```
  示波器通道      探头接到
  ──────────────────────────
  CH1 (黄)       SCLK
  CH2 (蓝)       MISO
  CH3 (紫)       CS（片选）
  CH4 (绿)       MOSI（可选）

  探头地线尽量短！长地线 = 地线环路 = 假抖动
```

### 2. 触发
- 触发源：CS 下降沿
- 时基：先看 8~16 个时钟周期的全貌，再放大到单 bit

### 3. 重点观察

```
  ┌────────────────────────────────────────────────────┐
  │  采样边沿到来时 MISO 的状态                          │
  │                                                    │
  │  ✅ 好的情况：                                      │
  │     SCLK ↑ 采样时，MISO 已经稳定在高或低              │
  │     setup time 充足，hold time 充足                  │
  │                                                    │
  │  ❌ 坏的情况：                                      │
  │     SCLK ↑ 采样时，MISO 正在翻转                     │
  │     采样点压在数据翻转附近 → 采到邻位的值              │
  │     → 表现为"整帧 1bit 偏移"                        │
  └────────────────────────────────────────────────────┘
```

### 4. mode0 vs mode1 波形判据
- **mode0 异常**：第一个上升沿采样时，MISO 还在变
- **mode1 正常**：采样点后移半周期，MISO 在采样瞬间已稳定

### 5. 软硬 A/B 对比（建议固定做）
```
同一固件下分别跑 mode0 / mode1，各采 1~2 分钟数据：
  统计：① 帧头命中率    ② 帧间距一致性    ③ 帧尾命中率
  谁稳定就选谁。
```

### 6. 常见误判排除
```
  排查优先级：
  ① 帧长变更（264 vs 268）导致"伪偏移"    → 用 radar_dump2 检查
  ② 字节序问题（int16 高低字节反了）        → 看 IQ 值是否在合理范围
  ③ 飞线/电平转换器造成边沿劣化             → 缩短走线，换探头重测
```

## Phase 4（规划）：雷达芯片自带 DSP 时的架构升级

> 当雷达芯片内部已经做完 1D-FFT / 2D-FFT，不再输出原始 IQ，
> 而是输出 FFT 结果矩阵时，整条信号处理链路需要重新规划。

### 4.1 先搞清楚：芯片输出的 1D-FFT 和 2D-FFT 到底是什么

```
  ┌───────────────────────────────────────────────────────────────────┐
  │                    FMCW 雷达信号处理的三级 FFT                     │
  └───────────────────────────────────────────────────────────────────┘

  原始 IQ 采样                                            你的芯片在这里
        │                                                 已经帮你做完了
        ▼                                                      │
  ┌──────────┐         ┌──────────────┐         ┌──────────┐   │
  │ 1D-FFT   │────────▶│   2D-FFT     │────────▶│ 3D-FFT   │   │
  │ Range FFT│         │ Doppler FFT  │         │ Angle FFT│   │
  │ 距离维   │         │ 速度维       │         │ 角度维   │   │
  └──────────┘         └──────────────┘         └──────────┘   │
       ↓                     ↓                       ↓         │
  [N_range个bin]        [N_range × N_chirp]     [需要多天线]    │
  每个bin→一个距离      行=距离 列=速度          这个芯片可能    │
                       交叉点→(距离,速度)       没做，需要你做   ▼
```

#### 1D-FFT 输出（Range FFT，距离维）

芯片对**每个 chirp** 的采样做 FFT，输出一个长度为 `N_range` 的复数数组。

```
  1D-FFT 输出：
  ┌─────┬─────┬─────┬─────┬─────┬─────┬───┬──────┐
  │bin 0│bin 1│bin 2│bin 3│bin 4│bin 5│...│bin N-1│
  └─────┴─────┴─────┴─────┴─────┴─────┴───┴──────┘
    DC    2.5m   5m   7.5m  10m  12.5m      最远

  每个 bin 是一个复数：幅度 = 信号强度，相位 = 后续测速用
```

**距离换算公式**：

```
  distance = bin_index × (c / (2 × BW))
           = bin_index × range_resolution

  例：BW = 60MHz → range_res = 3e8 / (2 × 60e6) = 2.5m
      bin 4 → 距离 = 4 × 2.5 = 10.0m
```

> 这一步和你现在的中间件 `RadarProcessor` 的 FFT 是**等价的**。
> 区别是：你现在是软件做 FFT，升级后芯片硬件帮你做了，**你不需要再做 FFT**。

#### 2D-FFT 输出（Range-Doppler Map，距离-速度二维图）

芯片收集 `N_chirp` 个 chirp 的 1D-FFT 结果，按**慢时间维度**再做一次 FFT。

```
  ┌────────────────────────────────────────────────────────┐
  │           Range-Doppler Map (2D-FFT 输出)               │
  │                                                        │
  │  速度 ↑                                                │
  │  bin   │                                               │
  │   7    │          ·                                    │
  │   6    │                                               │
  │   5    │     ·                                         │
  │   4    │                                               │
  │   3    │                    ★ (距离bin=4, 速度bin=3)    │
  │   2    │                                               │
  │   1    │  ·                                            │
  │   0    │──────────────────────────────────▶ 距离 bin    │
  │        0    1    2    3    4    5    ...   N_range      │
  │                                                        │
  │  ★ = 交叉点有能量 → 那个(距离,速度)上存在目标            │
  │  · = 噪声/杂波                                         │
  └────────────────────────────────────────────────────────┘
```

**速度换算公式**：

```
  velocity = doppler_bin × velocity_resolution

  velocity_resolution = λ / (2 × N_chirp × T_chirp)

  其中：
    λ = c / fc                    载波波长
    N_chirp                       一帧内 chirp 数量
    T_chirp                       单个 chirp 的持续时间

  注意：doppler_bin 需要做 fftshift！
    原始输出 bin 0 ~ N_chirp-1
    fftshift 后 bin 范围变成 -N_chirp/2 ~ N_chirp/2-1
    负 bin → 目标靠近（负速度）
    正 bin → 目标远离（正速度）
```

> 你现在的中间件用的是"帧间相位差"来测速，**精度较低且只能测很慢的速度**。
> 2D-FFT 方案精度高得多，能测的速度范围也大得多。

#### 角度怎么来？

```
  ┌────────────────────────────────────────────────────────┐
  │                    角度测量方案                          │
  │                                                        │
  │  方案 A：多天线 + 3D-FFT（需要硬件支持）                  │
  │  ──────────────────────────────────────                │
  │  雷达有 N_rx 个接收天线（如 4 个）                       │
  │  对同一距离-速度 bin，取 N_rx 个天线的复数值              │
  │  做第三维 FFT → 角度谱                                  │
  │                                                        │
  │  angle = arcsin(angle_bin × λ / (N_rx × d_antenna))    │
  │  d_antenna = 天线间距（通常 λ/2）                       │
  │                                                        │
  │  方案 B：单天线 → 无法直接测角                           │
  │  ──────────────────────────────────────                │
  │  如果芯片只有 1 个 RX 天线，物理上测不了角度              │
  │  只能靠机械旋转（舵机带动雷达转）来模拟扫描角度           │
  │  你现在的 sim_angle 就是这个思路                         │
  │                                                        │
  │  方案 C：MIMO 虚拟阵列                                  │
  │  ──────────────────────────────────────                │
  │  2TX × 4RX → 等效 8 元虚拟阵列                         │
  │  角度分辨率提升 4 倍                                    │
  │  需要芯片支持 MIMO 模式                                 │
  └────────────────────────────────────────────────────────┘
```

### 4.2 拿到 1D/2D-FFT 数据后，你还需要做什么

```
  芯片输出              你需要做的               最终结果
  ──────────────────────────────────────────────────────────

  情况 1：芯片只输出 1D-FFT（每 chirp 一组 range bin）
  ┌──────────┐     ┌────────────────────┐     ┌──────────┐
  │ 1D-FFT   │────▶│ ① CFAR 检测距离     │────▶│ 距离     │
  │ N个复数   │     │ ② 攒多chirp做2D-FFT │     │ 速度     │
  │          │     │ ③ CFAR 检测速度     │     │ (角度)   │
  └──────────┘     │ ④ 多天线→角度FFT    │     └──────────┘
                   └────────────────────┘

  情况 2：芯片输出 2D-FFT（Range-Doppler Map）
  ┌──────────┐     ┌────────────────────┐     ┌──────────┐
  │ 2D-FFT   │────▶│ ① 2D-CFAR 检测      │────▶│ 距离     │
  │ N×M矩阵  │     │ ② bin→物理量换算    │     │ 速度     │
  │          │     │ ③ 多天线→角度FFT    │     │ (角度)   │
  └──────────┘     └────────────────────┘     └──────────┘

  情况 3：芯片连 CFAR 也做完了（最省事）
  ┌──────────┐     ┌────────────────────┐     ┌──────────┐
  │ 目标列表  │────▶│ ① bin→物理量换算    │────▶│ 距离     │
  │ (peak list)│   │ ② 多天线→角度       │     │ 速度     │
  │          │     │ ③ 跟踪/滤波(可选)   │     │ 角度     │
  └──────────┘     └────────────────────┘     └──────────┘
```

> **核心结论**：芯片做的越多，STM32 需要做的越少。
> **但至少需要做**：bin 索引到物理量（米、m/s、度）的换算 + 目标筛选。

### 4.3 STM32F429 跑这个中间件，行不行？

#### 资源评估

```
  ┌──────────────────────────────────────────────────────────┐
  │              STM32F429 资源 vs 算法需求                    │
  │                                                          │
  │  CPU：Cortex-M4 @ 180MHz + 单精度 FPU + DSP 指令集       │
  │  RAM：256KB SRAM（部分型号有外部 SDRAM）                  │
  │  Flash：2MB                                              │
  │                                                          │
  │  ───────────────────────────────────────────              │
  │  算法需求估算：                                           │
  │                                                          │
  │  ● 如果只做 CFAR + 换算（芯片已做完 2D-FFT）：            │
  │    RAM: Range-Doppler Map 64×64×8B = 32KB  ✅ 够用       │
  │    CPU: 遍历矩阵做 CFAR，180MHz 轻松搞定  ✅             │
  │    → 完全没问题                                          │
  │                                                          │
  │  ● 如果需要自己做 FFT（芯片只给原始 IQ）：                 │
  │    64 点 FFT: CMSIS-DSP 库 <0.1ms            ✅          │
  │    256 点 FFT: CMSIS-DSP 库 ~0.5ms            ✅          │
  │    1024 点 FFT: ~3ms                          ✅ 还行     │
  │    2D-FFT 64×64: ~5ms                         ✅ 刚好     │
  │    2D-FFT 256×256: ~80ms                      ⚠️ 吃力    │
  │                                                          │
  │  ● 如果要做 3D-FFT（角度）：                              │
  │    取决于天线数，4天线额外加 ~1ms               ✅          │
  │    8天线额外加 ~3ms                             ✅          │
  │                                                          │
  │  结论：只要 FFT 点数不超过 256×128，STM32F429 都能实时     │
  └──────────────────────────────────────────────────────────┘
```

#### STM32 上的关键库

```
  CMSIS-DSP 库（ARM 官方，硬件加速）：
  ├── arm_cfft_f32()        复数 FFT（用这个！）
  ├── arm_cmplx_mag_f32()   复数取模（求幅度）
  ├── arm_max_f32()         找峰值
  └── arm_mean_f32()        求均值（CFAR 用）

  不要用 std::complex / std::cos 这种 C++ 标准库！
  用 CMSIS-DSP 能快 5-10 倍（它底层用 Cortex-M4 的 DSP 指令）
```

```
  │                        双 SPI 架构总览                             │
  │                                                                   │
  │  ┌──────────┐   SPI1 (Master)   ┌──────────┐   SPI2 (Slave)   ┌─────────┐
  │  │          │──────────────────▶│          │──────────────────▶│         │
  │  │  雷达    │   SCLK/MISO/CS    │ STM32    │   SCLK/MISO/CS   │ RK3588  │
  │  │  芯片    │◀──────────────────│ F429     │◀──────────────────│ (Linux) │
  │  │          │   读 FFT 数据      │          │   发目标结果      │         │
  │  └──────────┘                   └──────────┘                   └─────────┘
  │                                      │                              │
  │                                 ┌────┴────┐                    ┌────┴────┐
  │                                 │ 跑算法   │                    │ 显示/   │
  │                                 │ CFAR     │                    │ 决策/   │
  │                                 │ 换算     │                    │ 记录    │
  │                                 │ 分类     │                    │         │
  │                                 └─────────┘                    └─────────┘
  │                                                                   │
  │  数据流量对比：                                                     │
  │  SPI1: 雷达→STM32  原始 FFT 数据  ~几KB/帧   (大流量)              │

```

#### 为什么要这么做？（对比现有方案的优势）

```

  │              旧方案 vs 新方案 对比                              │
  │                                                              │
  │  旧方案（现在的）：                                            │
  │  雷达 ──IQ──▶ STM32 ──IQ──▶ RK3588 ──FFT+CFAR──▶ 目标       │
  │               打包264B       Linux驱动    Linux中间件          │

  │                                                              │

  │                                                              │
  │  新方案：                                                     │
  │  雷达 ──FFT──▶ STM32 ──CFAR+换算──▶ 目标结构体               │
  │    芯片内DSP     SPI1读    裸机跑算法       │                  │
  │                                            ▼                  │
  │                                   STM32 ──SPI2──▶ RK3588     │

  │                                                              │
  │  优势：                                                       │
  │  ① STM32 裸机/RTOS，确定性延迟 <1ms                          │
  │  ② SPI2 每帧只传几十字节（vs 264B）—— 带宽降 10 倍            │
  │  ③ RK3588 不再做 FFT，只管显示/决策                           │
  │  ④ 雷达芯片换了也不影响 Linux 端代码                          │
  └──────────────────────────────────────────────────────────────┘
```

```
  │       │  │  读取目标数据      │                           │
  │       │  │  CS 拉高 ◀────────│ 完成                      │
  │       │  │                   │                           │

  │  方案 B：轮询（简单但浪费）                                │
  │  ──────────────────────────────────────                  │
  │  RK3588 每 20ms 主动读一次 SPI                            │
  │  STM32 在 SPI buffer 里放最新一帧的结果                    │
  │  没有新数据时 target_count = 0                            │
  │                                                          │
  │  推荐方案 A，因为：                                       │
  │  ① 不浪费 CPU（只在有数据时才读）                          │
  │  ② 延迟最低（STM32 处理完立刻通知）                        │
  │  ③ Linux 端可以用 GPIO 中断驱动（比 hrtimer 省电）         │
  └──────────────────────────────────────────────────────────┘
```

### 4.5 STM32F429 端算法伪代码

```c
/* ==================== STM32 端主循环（裸机 / FreeRTOS） ==================== */

#include "arm_math.h"    /* CMSIS-DSP */

#define N_RANGE     64   /* 距离维 bin 数 */
#define N_DOPPLER   64   /* 速度维 chirp 数（具体看芯片配置） */
#define RANGE_RES   2.5f /* 距离分辨率 m */
#define VEL_RES     0.5f /* 速度分辨率 m/s（需根据芯片参数算） */
#define MAX_TARGETS 8

/* 目标结构体 */
typedef struct {
    uint16_t range_cm;      /* 距离，厘米 */
    int16_t  velocity_cmps; /* 速度，厘米/秒 */
    int16_t  angle_deg10;   /* 角度，0.1度 */
    uint16_t amplitude;     /* 幅度 */
} target_t;

/* 全局结果缓冲区 */
volatile target_t targets[MAX_TARGETS];
volatile uint8_t  target_count = 0;
volatile uint8_t  frame_id = 0;

void process_radar_frame(void)
{
    /* ① 从雷达芯片 SPI1 读取 2D-FFT 数据 */
    float32_t rd_map[N_RANGE * N_DOPPLER * 2];  /* 复数：实部+虚部交替 */
    spi1_read_rdmap(rd_map);                     /* 硬件相关：读 SPI1 */

    /* ② 求幅度矩阵 */
    float32_t magnitude[N_RANGE * N_DOPPLER];
    for (int i = 0; i < N_RANGE * N_DOPPLER; i++) {
        float32_t re = rd_map[2*i];
        float32_t im = rd_map[2*i + 1];
        magnitude[i] = sqrtf(re*re + im*im);  /* 或 arm_cmplx_mag_f32 */
    }

    /* ③ 2D-CFAR 检测 */
    target_count = 0;
    for (int r = 2; r < N_RANGE/2; r++) {       /* 跳过 DC，只看前半 */
        for (int d = 0; d < N_DOPPLER; d++) {
            float32_t cell = magnitude[r * N_DOPPLER + d];
            float32_t noise = estimate_noise_2d(magnitude, r, d);
            if (cell > noise * CFAR_THRESHOLD && target_count < MAX_TARGETS) {
                /* doppler bin 做 fftshift */
                int d_shifted = (d < N_DOPPLER/2) ? d : d - N_DOPPLER;
                targets[target_count].range_cm     = (uint16_t)(r * RANGE_RES * 100);
                targets[target_count].velocity_cmps = (int16_t)(d_shifted * VEL_RES * 100);
                targets[target_count].angle_deg10   = 0;  /* 单天线无角度 */
                targets[target_count].amplitude     = (uint16_t)cell;
                target_count++;
            }
        }
    }
    frame_id++;

    /* ④ 拉低 DRDY 通知 RK3588 来取数据 */
    HAL_GPIO_WritePin(DRDY_GPIO_Port, DRDY_Pin, GPIO_PIN_RESET);
}

/* SPI2 从机中断：RK3588 来读数据时触发 */
void HAL_SPI_TxCpltCallback(SPI_HandleTypeDef *hspi)
{
    if (hspi == &hspi2) {
        /* 传输完成，拉高 DRDY */
        HAL_GPIO_WritePin(DRDY_GPIO_Port, DRDY_Pin, GPIO_PIN_SET);
    }
}
```

### 核心逻辑

Phase 4 方案在**产品层面**是更优的工程决策——低延迟、省 CPU、解耦合。但秋招不是做产品，**你要的是技术深度的展示面**。

现在的方案让你一个人覆盖了：
1. **内核驱动**：hrtimer + workqueue + ring buffer + misc device + poll —— 这些是嵌入式 Linux 岗位的硬通货
2. **并发同步**：spinlock 保护中断上下文、mutex 保护进程上下文、atomic 解决裸读竞争 —— 能和面试官聊 race condition
3. **信号处理**：Hann 窗 + FFT + CFAR + 帧间测速 —— 说明你不是只会写驱动，还懂算法
4. **debug 方法论**：SPI mode 排查、5 个递进式调试工具 —— 面试最喜欢的"你遇到过什么难题"

如果 STM32 把活全干了，Linux 端就变成"GPIO 中断来了 → 读 70 字节 → 丢给 Qt"，三句话讲完，没东西可挖。
