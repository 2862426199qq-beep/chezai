## 第一章：网络通信的本质 — 两台电脑怎么说话

### 1.1 类比：打电话

```
你想和朋友通话，需要什么？

1. 你有电话机        →  你的程序有 socket（网络"插座"）
2. 你有电话号码      →  你的 IP 地址（如 192.168.137.233）
3. 你有分机号        →  端口号（如 8080）
4. 你拨号            →  connect()
5. 朋友接听          →  accept()
6. 你们说话          →  write() / read()
7. 挂断              →  close()
```

### 1.2 IP 地址和端口号

```
IP 地址：标识"哪台机器"
  192.168.137.233 ← 你的开发板
  192.168.137.1   ← 你的 PC

端口号：标识"这台机器上的哪个程序"
  一台机器可以同时运行很多网络程序，端口号区分它们
  
  常见端口：
    80   → HTTP 网页
    443  → HTTPS
    22   → SSH（你用 MobaXterm 连板子就是这个）
    5902 → VNC（你看 Qt 界面用的）
    8080 → 我们的 MJPEG 推流服务器

  同一个端口同时只能被一个程序占用
```

### 1.3 TCP 和 UDP — 两种通信方式

```
TCP（传输控制协议）— 打电话：
  ✅ 可靠：数据保证到达，保证顺序
  ✅ 有连接：先建立连接，再传数据
  ❌ 慢一点：每个包都要确认（ACK）
  用途：网页、文件传输、远程控制、MJPEG 推流（我们用的）

UDP（用户数据报协议）— 发短信：
  ✅ 快：发完不等回复
  ❌ 不可靠：可能丢包、乱序
  ❌ 无连接：直接发，不管对方在不在
  用途：视频通话、游戏、DNS查询
```

**我们选 TCP** 的原因：MJPEG 的每一帧 JPEG 图片必须完整到达浏览器才能显示，丢了一半的 JPEG 是乱码。

## 第二章：TCP 通信的完整流程 — socket 编程

### 2.1 服务端（Server）和客户端（Client）

```
我们的场景：
  服务端 = 开发板上的 VehicleTerminal（推视频流）
  客户端 = 你 PC 上的浏览器（看视频流）

角色分工：
  服务端：等待别人来连接（被动）
  客户端：主动发起连接
```

### 2.2 Linux C 语言的 socket API（底层原理）

虽然我们用 Qt 封装了，但面试要知道底层：

```c
// ==================== 服务端流程 ====================

// 第 1 步：创建 socket（电话机）
int sockfd = socket(AF_INET,      // IPv4 协议
                    SOCK_STREAM,   // TCP 类型（流式）
                    0);            // 自动选协议

// 第 2 步：绑定地址（告诉系统"我在 8080 端口等电话"）
struct sockaddr_in addr;
addr.sin_family      = AF_INET;
addr.sin_port        = htons(8080);        // 端口号（htons: 主机字节序→网络字节序）
addr.sin_addr.s_addr = INADDR_ANY;         // 监听所有网卡（0.0.0.0）

bind(sockfd, (struct sockaddr*)&addr, sizeof(addr));

// 第 3 步：开始监听（电话响铃状态）
listen(sockfd, 5);   // 5 = 最多排队 5 个等待连接的客户端

// 第 4 步：接受连接（接电话）—— 阻塞，等客户端来
int clientfd = accept(sockfd, NULL, NULL);
// 返回一个新的 fd，专门和这个客户端通信
// 原来的 sockfd 继续监听新客户端

// 第 5 步：收发数据
write(clientfd, "Hello", 5);       // 给客户端发数据
char buf[1024];
int n = read(clientfd, buf, 1024); // 从客户端收数据（阻塞）

// 第 6 步：关闭
close(clientfd);  // 关闭和这个客户端的连接
close(sockfd);    // 关闭监听 socket
```

```c
// ==================== 客户端流程 ====================

int sockfd = socket(AF_INET, SOCK_STREAM, 0);

struct sockaddr_in server_addr;
server_addr.sin_family      = AF_INET;
server_addr.sin_port        = htons(8080);
inet_pton(AF_INET, "192.168.137.233", &server_addr.sin_addr);

// 主动连接服务端（拨号）
connect(sockfd, (struct sockaddr*)&server_addr, sizeof(server_addr));

// 收发数据
char buf[1024];
int n = read(sockfd, buf, 1024);   // 读服务端发来的数据

close(sockfd);
```

### 2.3 整体流程图

```
        服务端                              客户端
        ──────                              ──────
    socket()                            socket()
       │                                   │
    bind(8080)                              │
       │                                   │
    listen()                                │
       │                                   │
    accept() ◀──── TCP 三次握手 ────▶ connect()
       │          SYN → SYN+ACK → ACK       │
       │                                   │
    write() ──────── 数据 ────────▶ read()
       │                                   │
    read()  ◀──────── 数据 ──────── write()
       │                                   │
    close() ──── TCP 四次挥手 ────▶ close()


三次握手：    确认双向收发能力：
    第一次： 服务端确认了“客户端能发，我能收”。

    第二次： 客户端确认了“我能发也能收，服务端也能发也能收”。

    第三次： 服务端确认了“客户端也能收”。
    如果只有两次，服务端发完确认后并不知道客户端是否收到了，这会导致资源浪费。

四次挥手：    可靠断开连接：
    第一次： 主动断开的一方（如客户端）发 FIN，告诉对方“我要断了”。
    第二次： 被动断开的一方（如服务端）回复 ACK，确认收到了断开请求。
    第三次： 被动断开的一方发 FIN，告诉主动断开的一方“我也要断了”。
    第四次： 主动断开的一方回复 ACK，确认收到了断开请求。

```

### 2.4 关键概念速记

| 概念 | 说明 | 类比 |
|------|------|------|
| `socket` | 通信端点，返回文件描述符 fd | 电话机 |
| `bind` | 把 socket 绑定到 IP:端口 | 给电话机分配电话号码 |
| `listen` | 开始监听，允许别人连进来 | 电话开机，等来电 |
| `accept` | 接受一个客户端连接，返回新 fd | 接电话，开始通话 |
| `connect` | 客户端主动连接服务端 | 拨号 |
| `read/write` | 收发数据 | 说话/听话 |
| `close` | 关闭连接 | 挂电话 |

### 2.5 字节序：为什么要 htons()

```
问题：不同 CPU 存储多字节数字的方式不同

  端口号 8080 = 0x1F90

  大端序（网络标准）：高字节在前    → 内存中 [1F] [90]
  小端序（x86/ARM）：低字节在前     → 内存中 [90] [1F]

  如果双方字节序不同，8080 会被解读成 0x901F = 36895 → 连接到错误端口

htons() = Host TO Network Short（主机序 → 网络序，16位）
htonl() = Host TO Network Long（32位）
ntohs() = 反过来

网络传输统一用大端序，发送前转换，接收后转换
```

### 2.6 为什么 accept 返回新 fd？

```
如果只有一个 fd，那么：
  正在和客户端A通信时，客户端B来连接 → 没法处理

实际做法：
  sockfd（监听 fd）：专门等新连接，永远不传数据
  clientfd（连接 fd）：accept 返回的新 fd，专门和某个客户端通信

  类比：
  sockfd = 前台总机电话（只接新来电，转接后挂掉）
  clientfd = 分机电话（和客户一对一聊）
```

## 第三章：Qt 对 socket 的封装

### 3.1 Qt 为什么要封装？

原始 socket API 的痛点：
```
1. accept() / read() 是阻塞调用 → 需要手动管理多线程
2. 每次连接都要管 fd 的生命周期 → 容易忘 close 导致资源泄漏
3. 多客户端要用 select/poll/epoll → 代码量爆炸
4. 没有"有新连接来了"的通知机制 → 得自己轮询
```

Qt 封装后：
```
1. 用信号-槽替代阻塞 → newConnection()、readyRead()、disconnected() 信号
2. QTcpSocket 析构自动 close → RAII，不怕忘
3. 内部用事件循环 + 非阻塞 IO → 单线程就能管多个客户端
4. QTcpServer 自动 listen + accept → 你只关注业务逻辑
```

### 3.2 Qt 和 Linux C API 的对应关系

```
Linux C API                          Qt 封装
─────────────────────────────────────────────────────
socket() + bind() + listen()    →    QTcpServer::listen(QHostAddress::Any, 8080)
accept()                        →    QTcpServer::nextPendingConnection()
                                     （收到 newConnection() 信号后调用）
connect()                       →    QTcpSocket::connectToHost("192.168.137.233", 8080)
write(fd, data, len)            →    QTcpSocket::write(data)
read(fd, buf, len)              →    QTcpSocket::readAll()
                                     （收到 readyRead() 信号后调用）
close(fd)                       →    QTcpSocket::disconnectFromHost()
                                     或 delete socket
```

### 3.3 Qt TCP Server 最小示例

```cpp
// 最简单的 TCP 回声服务器：客户端发什么就回什么
#include <QTcpServer>
#include <QTcpSocket>

class EchoServer : public QObject {
    Q_OBJECT
public:
    EchoServer() {
        server = new QTcpServer(this);

        // 信号：有新客户端连接时触发
        connect(server, &QTcpServer::newConnection, this, [this]() {
            QTcpSocket *client = server->nextPendingConnection();

            // 信号：客户端发来数据时触发
            connect(client, &QTcpSocket::readyRead, client, [client]() {
                QByteArray data = client->readAll();  // 读取客户端发的数据
                client->write(data);                   // 原样回发
            });

            // 信号：客户端断开时触发
            connect(client, &QTcpSocket::disconnected, client, &QObject::deleteLater);
        });

        server->listen(QHostAddress::Any, 9999);  // 监听 9999 端口
        // 内部自动做了：socket() + bind() + listen()
    }

private:
    QTcpServer *server;
};
```

```
测试方法（在 PC 终端）：
  $ telnet 192.168.137.233 9999
  > hello               ← 你输入
  < hello               ← 服务器回显

  或者用 netcat：
  $ echo "test" | nc 192.168.137.233 9999
```

## 第四章：HTTP 协议基础 — 浏览器怎么和服务器通信

### 4.1 为什么需要 HTTP？

```
TCP 只管"可靠地传字节"，不管字节是什么意思。
HTTP 定义了"字节的格式和含义"——请求长什么样，响应长什么样。

浏览器只认 HTTP 协议。你用 TCP 发任意字节给浏览器，它不知道怎么处理。
但如果你按 HTTP 格式回复，浏览器就能正确显示。
```

### 4.2 HTTP 请求（浏览器 → 服务器）

```
当你在浏览器输入 http://192.168.137.233:8080 并回车，浏览器发出：

GET / HTTP/1.1\r\n                    ← 请求行：方法 路径 协议版本
Host: 192.168.137.233:8080\r\n        ← 请求头
User-Agent: Mozilla/5.0 ...\r\n       ← 浏览器标识
Accept: text/html,image/*\r\n         ← 我能接受的内容类型
\r\n                                   ← 空行，表示请求头结束

各部分含义：
  GET    → 请求方法（"给我数据"，不是"收我数据"）
  /      → 请求路径（根路径）
  HTTP/1.1 → 协议版本
  \r\n   → 回车换行（HTTP 用 \r\n 而不是 \n）
```

### 4.3 HTTP 响应（服务器 → 浏览器）

```
服务器回复：

HTTP/1.1 200 OK\r\n                         ← 状态行：版本 状态码 描述
Content-Type: text/html\r\n                  ← 告诉浏览器：这是网页
Content-Length: 13\r\n                        ← 内容长度（字节数）
\r\n                                          ← 空行，表示响应头结束
Hello, World!                                 ← 响应体（实际内容）

常见状态码：
  200 → OK（成功）
  404 → Not Found（找不到）
  502 → Bad Gateway（你之前打错 IP 看到的就是这个）
```

### 4.4 Content-Type — 告诉浏览器怎么处理数据

```
Content-Type 决定了浏览器用什么方式显示响应体：

text/html        → 当网页渲染（解析 HTML 标签）
text/plain       → 当纯文本显示
image/jpeg       → 当图片显示
image/png        → 当 PNG 图片显示
application/json → 当 JSON 数据处理

多部分类型（MJPEG 用的）：
multipart/x-mixed-replace;boundary=xxx
  → 特殊类型："我会连续发多个部分，每个部分用 --xxx 分隔"
  → 浏览器收到新部分就替换掉旧的（replace）
  → 效果：连续收到 JPEG → 不断替换 → 看起来就是视频！
```

## 第五章：MJPEG 协议 — 用 HTTP 推视频

### 5.1 什么是 MJPEG

```
MJPEG = Motion JPEG = 动态 JPEG

原理极其简单：
  每一帧都是一张独立的 JPEG 图片
  连续发 30 张/秒 → 人眼看起来就是视频

和 H.264/H.265 视频编码的区别：
  H.264：帧间压缩（参考前后帧）→ 体积小但编解码复杂
  MJPEG：帧内压缩（每帧独立）→ 体积大但编解码简单

  MJPEG 720p quality=70 → 单帧 ~36KB → 30fps → 约 1MB/s 带宽
  H.264 720p            → 约 200KB/s（压缩 5 倍），但需要编解码器
```

### 5.2 MJPEG over HTTP 的格式

```
服务端先发一次 HTTP 响应头（建立长连接）：

HTTP/1.1 200 OK\r\n
Content-Type: multipart/x-mixed-replace;boundary=mjpegframe\r\n
Cache-Control: no-cache\r\n
Connection: keep-alive\r\n
\r\n

然后不断发帧（每帧格式）：

--mjpegframe\r\n                        ← 边界标记（和响应头里的 boundary 一致）
Content-Type: image/jpeg\r\n            ← 这一部分是 JPEG 图片
Content-Length: 36802\r\n               ← 图片大小（字节）
\r\n                                     ← 空行
[36802 字节的 JPEG 二进制数据]           ← 图片本体
\r\n                                     ← 结尾换行

--mjpegframe\r\n                        ← 下一帧开始
Content-Type: image/jpeg\r\n
Content-Length: 37100\r\n
\r\n
[37100 字节的 JPEG 二进制数据]
\r\n

...无限循环...
```

### 5.3 浏览器怎么处理

```
浏览器看到 Content-Type: multipart/x-mixed-replace，就知道：
  1. 这是一个持续更新的流
  2. 每收到一个 --boundary 分隔的部分 → 替换掉上一个
  3. 每个部分是 image/jpeg → 当图片显示

效果：
  收到第 1 帧 JPEG → 显示
  收到第 2 帧 JPEG → 替换第 1 帧，显示第 2 帧
  收到第 3 帧 JPEG → 替换第 2 帧，显示第 3 帧
  ...
  30 帧/秒替换 → 人眼看到连续视频
```

## 第六章：逐行解读 mjpeg_server — 我们的代码怎么写的

### 6.1 头文件 `mjpeg_server.h`

```cpp
class MjpegServer : public QObject
{
    Q_OBJECT   // Qt 元对象系统，支持信号-槽

public:
    explicit MjpegServer(quint16 port = 8080, QObject *parent = nullptr);
    ~MjpegServer();

    bool isListening() const;    // 是否在监听
    quint16 serverPort() const;  // 监听端口号

public slots:
    // 核心接口：外部每来一帧就调这个，server 把帧推给所有客户端
    void onNewFrame(const QImage &img);

private slots:
    void onNewConnection();        // 有新客户端连接时触发
    void onClientDisconnected();   // 客户端断开时触发

private:
    QTcpServer *m_server;          // TCP 监听服务器（封装了 socket+bind+listen）
    QList<QTcpSocket*> m_clients;  // 所有已连接的客户端列表
    QMutex m_mutex;                // 保护 m_clients 的互斥锁
    int m_jpegQuality;             // JPEG 压缩质量（0-100）
};
```

**为什么需要 QMutex？**
```
onNewFrame() 在 UI 线程被调用（每帧触发）
onNewConnection() / onClientDisconnected() 也在 UI 线程

它们都访问 m_clients 列表。虽然当前都在 UI 线程（同一线程不会并发），
但加锁是防御性编程——如果将来把推流放到子线程，不加锁就会崩。
```

### 6.2 构造函数

```cpp
MjpegServer::MjpegServer(quint16 port, QObject *parent)
    : QObject(parent)
    , m_server(new QTcpServer(this))     // 创建 TCP 服务器
    , m_jpegQuality(70)
{
    // 当有新客户端连接时，Qt 发 newConnection 信号 → 我们的 onNewConnection 处理
    connect(m_server, &QTcpServer::newConnection,
            this, &MjpegServer::onNewConnection);

    // 开始监听：QHostAddress::Any = 0.0.0.0 = 所有网卡都接受连接
    if (m_server->listen(QHostAddress::Any, port)) {
        qDebug() << "MjpegServer: listening on port" << port;
    } else {
        qWarning() << "MjpegServer: failed to listen on port" << port;
    }
}
```

```
m_server->listen() 内部做了什么？
  等价于 Linux C 的：
    int fd = socket(AF_INET, SOCK_STREAM, 0);
    bind(fd, addr_8080, ...);
    listen(fd, backlog);
  然后把 fd 注册到 Qt 事件循环（内部用 epoll），
  当有新连接时触发 newConnection() 信号。

  你不需要手动管 fd、不需要写 epoll、不需要开线程 → 这就是 Qt 封装的价值。
```

### 6.3 处理新连接

```cpp
void MjpegServer::onNewConnection()
{
    while (m_server->hasPendingConnections()) {
        // 取出一个客户端连接（等价于 accept()）
        QTcpSocket *client = m_server->nextPendingConnection();
        if (!client) continue;

        // 浏览器连接后会发 HTTP GET 请求，我们不需要解析，直接丢弃
        connect(client, &QTcpSocket::readyRead, client, [client]() {
            client->readAll();  // 读出来就扔了
        });

        // 客户端断开时清理
        connect(client, &QTcpSocket::disconnected,
                this, &MjpegServer::onClientDisconnected);

        // 发送 HTTP 响应头 → 告诉浏览器："我要开始推 MJPEG 流了"
        client->write(HTTP_HEADER);
        client->flush();

        // 加入客户端列表
        QMutexLocker lock(&m_mutex);
        m_clients.append(client);
    }
}
```

```
发生了什么？

  浏览器输入 http://192.168.137.233:8080 回车
    │
    ├─ 浏览器内部：socket() → connect(192.168.137.233:8080)
    │  TCP 三次握手完成
    │
    ├─ 板子的 QTcpServer 收到连接 → 触发 newConnection() 信号
    │
    ├─ onNewConnection() 被调用
    │  ├─ nextPendingConnection() 拿到 QTcpSocket*
    │  ├─ 浏览器发来 "GET / HTTP/1.1\r\n..." → readyRead → 我们 readAll() 丢弃
    │  ├─ 我们回 "HTTP/1.1 200 OK\r\nContent-Type: multipart/x-mixed-replace..."
    │  └─ 浏览器收到响应头，准备接收连续图片
    │
    └─ 之后每帧 onNewFrame() 推 JPEG → 浏览器持续显示
```

### 6.4 推帧（核心）

```cpp
void MjpegServer::onNewFrame(const QImage &img)
{
    QMutexLocker lock(&m_mutex);
    if (m_clients.isEmpty()) return;   // ★ 没人看就不编码，省 CPU

    // ① 把 QImage 编码为 JPEG（720p quality=70 约 36KB，耗时 ~5-10ms）
    QByteArray jpeg;
    QBuffer buf(&jpeg);
    buf.open(QIODevice::WriteOnly);
    img.save(&buf, "JPEG", m_jpegQuality);
    buf.close();

    // ② 构造 multipart 帧头（告诉浏览器"这是一帧 JPEG，大小 xxx 字节"）
    QByteArray header;
    header.append("--mjpegframe\r\n");
    header.append("Content-Type: image/jpeg\r\n");
    header.append("Content-Length: " + QByteArray::number(jpeg.size()) + "\r\n");
    header.append("\r\n");

    // ③ 推给所有客户端
    QMutableListIterator<QTcpSocket*> it(m_clients);
    while (it.hasNext()) {
        QTcpSocket *client = it.next();

        // ★ 背压控制：如果客户端写缓冲区积压 > 1MB → 网速太慢，断开
        if (client->bytesToWrite() > 1024 * 1024) {
            client->disconnectFromHost();
            it.remove();
            client->deleteLater();
            continue;
        }

        // 发送：帧头 + JPEG 数据 + 换行
        client->write(header);
        client->write(jpeg);
        client->write("\r\n");
    }
}
```

```
数据流：

  CameraThread → frameReady(QImage) → MainWindow lambda
                                         │
                                ┌────────┴─────────┐
                                │                  │
                         setPixmap()          m_mjpegServer->onNewFrame(img)
                         UI 显示                    │
                                             ① QImage → JPEG 编码
                                             ② 构造 HTTP multipart 帧头
                                             ③ TCP write() 推给所有浏览器
```

### 6.5 背压控制 — 为什么要检查 bytesToWrite

```
问题场景：
  摄像头 30fps，每帧 JPEG ~36KB
  → 每秒推 ~1MB 数据

  如果某个客户端网速只有 500KB/s：
    每秒发 1MB，只能收 500KB → 差额 500KB 堆积在内核发送缓冲区
    10 秒后堆积 5MB → 内存持续增长
    100 秒后堆积 50MB → 系统内存告急

  更糟的是：Qt 的 write() 不是直接发，而是先写入 Qt 内部缓冲区，
  然后 Qt 事件循环异步发送。如果对方收不过来，缓冲区无限增长。

我们的解决方案：
  if (client->bytesToWrite() > 1024 * 1024)  // 缓冲区 > 1MB
    → 断开这个慢客户端，不让它拖垮整个服务

  bytesToWrite() 返回"已经调了 write() 但还没被内核发出去的字节数"
  这个值越大，说明对方越跟不上。

面试加分点：这就是"背压"（backpressure）的概念——
  生产者速度 > 消费者速度时，需要某种机制来保护生产者。
  我们的策略是"丢弃慢消费者"。
```

```
  ❌ 浏览器不认识 NV12 格式
  ❌ 没有任何浏览器/播放器支持裸 NV12 流
```

## 面试 Q&A — 网络推流部分（口语化回答）

### Q1：你这个远程监控是怎么做的？

> 我在 VehicleTerminal 里加了一个 **MJPEG over HTTP 的 TCP 推流服务**。
> 程序启动时用 `QTcpServer` 监听 8080 端口，浏览器连上来后，我先给它返回一个 HTTP 响应头，`Content-Type` 设成 `multipart/x-mixed-replace`，
> 然后每来一帧摄像头画面，就把 `QImage` 编码成 JPEG，按 multipart 格式 `write` 给所有已连接的客户端。
> 浏览器原生支持这种格式，打开网页就能直接看到实时视频流。

### Q2：为什么选 MJPEG 而不是 H.264/RTSP？

> 两个原因：
> **第一**，MJPEG 实现极简——每帧独立编码一张 JPEG 就行，不需要帧间参考、不需要引入 ffmpeg 这样的重量级编解码库，我 130 行代码就搞定了。
> **第二**，浏览器原生支持 `multipart/x-mixed-replace`，用户不需要安装任何播放器，打开网页就能看。如果用 RTSP 或 H.264，还得用 VLC 或写前端播放器。
> 当然 MJPEG 的代价是**带宽大**——720p 大约 1MB/s，H.264 同画质只要 200KB/s。但在局域网场景下，带宽不是瓶颈。

### Q3：底层的 socket 流程你了解吗？

> Qt 的 `QTcpServer::listen()` 内部封装了 `socket()` 创建套接字、`bind()` 绑定到指定端口、`listen()` 进入监听状态这三步。
> 当有客户端连进来，`QTcpServer` 触发 `newConnection` 信号，我在槽函数里调 `nextPendingConnection()` 拿到 `QTcpSocket` 指针——这一步对应底层的 `accept()`，返回一个新的 socket 专门和这个客户端通信。
> 之后就是 `write()` 发数据、`readAll()` 收数据，最后 `disconnectFromHost()` 关闭连接。
> Qt 底层用 **epoll + 事件循环**实现非阻塞 IO，所以我不需要手动管线程或 select。

### Q4：怎么处理多个客户端同时看？

> 我维护一个 `QList<QTcpSocket*>` 存所有已连接的客户端指针。
> 每帧的 JPEG **只编码一次**，然后遍历列表 `write` 给每个客户端。新客户端连上就 `append`，断开就 `removeAll` + `deleteLater`。
> 关键是加了**背压控制**——每次推帧前检查 `bytesToWrite()`，如果某个客户端的写缓冲区积压超过 1MB，说明它网速跟不上，就主动断开它，防止慢客户端拖垮整个服务。

### Q5：什么是背压？你怎么实现的？

> 背压是指**生产者速度超过消费者速度**时的保护机制。
> 我们的场景：摄像头每秒产 30 帧，每帧 JPEG 约 36KB，总共约 1MB/s。如果某个客户端只有 500KB/s 的带宽，差额就会堆积在 TCP 发送缓冲区里。
> 我用 `QTcpSocket::bytesToWrite()` 检测积压量——这个函数返回"已经调了 write 但还没被内核发出去"的字节数。超过 1MB 就断开这个客户端，保护服务端内存不被撑爆。

### Q6：JPEG 编码在哪个线程做的？会不会卡 UI？

> 目前在 **UI 线程**做。CameraThread 的 `frameReady` 信号通过 Qt 信号槽投递到 UI 线程，在 lambda 回调里先 `setPixmap` 显示，再调 `onNewFrame` 做 JPEG 编码和推流。
> 720p quality=70 的 JPEG 编码大约 5-10ms，Qt 界面刷新周期是 16ms（60fps），在余量之内不会卡。
> 如果要支持更高分辨率或更多客户端，可以把编码挪到独立线程，但当前场景不需要。

### Q7：没有客户端观看时有性能开销吗？

> 没有。`onNewFrame()` 第一行就检查 `m_clients.isEmpty()`，如果没人连着直接 return，**不做 JPEG 编码**。所以关闭浏览器后 CPU 占用和没加这个功能时完全一样。

### Q8：你说比 VNC 延迟低，为什么？

> VNC 传的是**整个桌面窗口的变化区域**，编码方式是帧差 + RLE/ZRLE 压缩，还要处理鼠标、按钮、文字等 UI 元素的重绘。
> 而 MJPEG 只传**摄像头画面本身**，一帧就是一张独立的 JPEG，编码和解码都极其简单。
> 实测下来 MJPEG 从采集到浏览器显示比 VNC 快了可感知的一截，尤其是画面变化大的时候（VNC 帧差大 → 编码慢）。

### Q9：TCP 三次握手能简单说说吗？

> 客户端发 **SYN** 给服务端（"我要连接"），服务端回 **SYN+ACK**（"收到，我也准备好了"），客户端再回 **ACK**（"我确认收到了你的回复"）。
> 为什么不能两次？两次的话服务端不知道客户端有没有收到 SYN+ACK——可能 SYN+ACK 丢了，服务端以为连接建好了，客户端却没收到。三次确保**双方都确认了对方的收发能力**。

### Q10：如果让你优化，下一步会怎么做？

> 三个方向：
> 1. **JPEG 编码移到子线程**——避免客户端多的时候阻塞 UI 线程；
> 2. **自适应帧率**——根据客户端的 `bytesToWrite()` 动态降帧率而不是直接断开；
> 3. **切换 H.264 + WebSocket**——如果带宽是瓶颈，可以用 RK3588 的硬件 H.264 编码器（MPP），压缩率比 MJPEG 高 5 倍，但实现复杂度也高很多。

# RTMP 推流笔记 — MPP 硬件 H.264 编码 + ffmpeg RTMP

> 平台：RK3588S LubanCat-4 | Qt 5.12.8 | 项目：VehicleTerminal
> 目标：将摄像头 NV12 帧通过 MPP 硬件编码为 H.264，再通过 RTMP 协议推送到云端

---

## 第一章：为什么要做 RTMP 推流

### 1.1 MJPEG vs RTMP+H.264 对比

```
我们已经有了 MJPEG 推流（mjpeg_server），它在局域网下非常好用。
但如果要推到"云端"让远程客户查看，MJPEG 的带宽问题就是致命的：

  MJPEG 720p quality=70：每帧 ~36KB × 30fps ≈ 1MB/s = 8Mbps
  H.264 720p 2Mbps CBR ：每秒 ~250KB                 = 2Mbps

  差 4 倍！

  家庭/4G 上行带宽通常只有 2-5Mbps，MJPEG 推不出去。
  H.264 只要 2Mbps，刚好能推。
```

| 维度 | MJPEG over HTTP | RTMP + H.264 |
|------|----------------|--------------|
| **压缩方式** | 帧内压缩（每帧独立 JPEG） | 帧间压缩（I/P/B 帧参考） |
| **带宽** | ~8Mbps (720p) | ~2Mbps (720p) |
| **编码器** | Qt 软件 JPEG (CPU ~5ms) | MPP 硬件 H.264 (CPU ~0) |
| **延迟** | 极低（无帧间依赖） | 较低（GOP 内有参考帧延迟） |
| **浏览器** | 原生支持 ✅ | 需 flv.js 或 hls.js |
| **服务端** | 不需要，自己就是 server | 需要 RTMP 服务器（nginx-rtmp） |
| **适合场景** | 局域网调试、嵌入式预览 | 公网推流、云端监控、多人观看 |

### 1.2 结论：互补，不是替代

```
局域网调试 → 用 MJPEG（简单、浏览器直接看、延迟最低）
推到云端   → 用 RTMP+H.264（省带宽、硬编不占 CPU、行业标准）

两个同时保留，MJPEG 通过 frameReady(QImage) 驱动，
RTMP 通过 nv12FrameReady(void*, int) 驱动，互不干扰。
```

## 第二章：H.264 编码基础 — 为什么比 JPEG 小 4 倍

### 2.1 关键概念：帧间预测

```
JPEG/MJPEG 的思路：
  每帧独立压缩（帧内压缩）
  一个人站着不动 → 每帧都存完整的背景 + 人 → 浪费

H.264 的思路：
  第 1 帧（I 帧）：完整压缩（和 JPEG 类似，但用更高级的算法）
  第 2~N 帧（P 帧）：只存"和前一帧的差异"
  人站着不动 → 90% 的像素没变 → 只存 10% 变化部分 → 体积极小

  I 帧（Intra）：关键帧，完整图像，体积大（~50KB）
  P 帧（Predicted）：预测帧，只有差异，体积小（~5KB）

  GOP（Group of Pictures）：一组帧，从一个 I 帧开始到下一个 I 帧之前
  GOP = 50 帧 → 每 50 帧才有一个大的 I 帧，其余 49 帧都是小 P 帧
```

### 2.2 为什么需要硬件编码器

```
H.264 编码算法非常复杂：
  - 宏块划分（16×16 / 8×8 / 4×4）
  - 运动估计（搜索前后帧中最相似的块）
  - 变换量化（DCT + 量化）
  - CABAC 熵编码（算术编码）

  720p CPU 软编码（libx264）：
    ARM Cortex-A55 单核约 30-50ms/帧 → 只能编 20fps，CPU 占用 100%

  720p MPP 硬编码（RK3588 VENC）：
    硬件 IP 核，~1-2ms/帧，CPU 占用 ~0%
    RK3588 支持 8K@30fps H.265 或 4K@120fps H.264 硬编码
```

### 2.3 SPS 和 PPS — 解码器必须知道的"说明书"

```
SPS（Sequence Parameter Set）：
  告诉解码器：分辨率多少、Profile 是什么、级别多少
  类比：一本书的"目录页"

PPS（Picture Parameter Set）：
  告诉解码器：每帧怎么分块、用什么熵编码
  类比：一本书的"阅读指南"

RTMP 推流时，SPS/PPS 必须在第一个视频包之前发送（写入 extradata），
否则播放器收到 P 帧也不知道怎么解码。

我们的代码在 initMppEncoder() 中用 MPP_ENC_GET_HDR_SYNC 获取 SPS/PPS，
然后设到 AVStream 的 codecpar->extradata 里。
```

## 第三章：RTMP 协议 — 从板子到云端的通道

### 3.1 什么是 RTMP

```
RTMP = Real-Time Messaging Protocol（实时消息传递协议）

  最初由 Adobe 为 Flash 设计（2008 年左右）
  虽然 Flash 已死，但 RTMP 推流至今仍是行业标准

  工作模式：
    推流端（我们） ──TCP──▶ RTMP 服务器 ──HTTP-FLV/HLS──▶ 观看端

  底层是 TCP 长连接，数据封装在 FLV 容器里
  我们用 ffmpeg 的 libavformat 做 FLV 封装 + RTMP 推送
```

### 3.2 RTMP 推流架构

```
┌──────────── 开发板 ────────────┐      ┌─── RTMP 服务器 ───┐     ┌── 观看端 ──┐
│                                │      │                    │     │            │
│  V4L2 NV12 → MPP H.264 编码   │      │  nginx-rtmp        │     │  VLC       │
│       │                        │      │  或 SRS            │     │  flv.js    │
│  ffmpeg libavformat            │      │  或云服务          │     │  ffplay    │
│       │                        │      │                    │     │            │
│  RTMP push ──── TCP ───────────┼──▶──┤  接收 → 分发 ──────┼──▶──┤  拉流播放  │
│                                │      │                    │     │            │
│  rtmp://server/live/camera     │      │  rtmp://server/... │     │            │
└────────────────────────────────┘      └────────────────────┘     └────────────┘

推流URL格式：rtmp://服务器IP/应用名/流名
  例：rtmp://192.168.137.1/live/camera
       │         │          │      │
    协议     服务器IP    应用名  流名（自定义）
```

### 3.3 RTMP vs HTTP-FLV vs HLS

```
三种都是"从 RTMP 服务器拉流"的方式，区别在延迟和兼容性：

  RTMP 拉流：VLC/ffplay 直接打开 rtmp:// 地址
    延迟：1-3秒
    缺点：浏览器不支持（Flash 已死）

  HTTP-FLV 拉流：nginx-rtmp 自动转成 http://server/live/camera.flv
    延迟：1-3秒
    优点：配合 flv.js 可在浏览器播放

  HLS 拉流：nginx-rtmp 自动切成 .m3u8 + .ts 切片
    延迟：5-30秒（切片缓冲，延迟最大）
    优点：所有浏览器原生支持（video 标签）

我们推 RTMP，服务端自动转 HTTP-FLV 和 HLS → 覆盖所有观看端
```

## 第四章：MPP 编码器 — RK3588 的硬件加速

### 4.1 MPP 是什么

```
MPP = Media Process Platform（媒体处理平台）
Rockchip 提供的用户态硬件编解码 SDK

  内核驱动：/dev/mpp_service（或 /dev/video-enc0）
  用户态库：librockchip_mpp.so
  头文件：/usr/include/rockchip/rk_mpi.h

MPP 和 V4L2 类似，也是"把数据喂给硬件，拿回结果"的模式：
  V4L2：  管理摄像头采集（input:sensor → output:NV12）
  MPP：   管理视频编码（input:NV12 → output:H.264 NAL）
```

### 4.2 MPP 编码 API 完整流程

```
// ===== 初始化阶段 =====

mpp_create(&ctx, &mpi)              // 创建上下文和 API 句柄
    │                                  类比：V4L2 的 open("/dev/video11")
    ▼
mpp_init(ctx, MPP_CTX_ENC,          // 初始化为编码器，编码类型 H.264
         MPP_VIDEO_CodingAVC)         类比：V4L2 的 VIDIOC_S_FMT
    │
    ▼
mpp_enc_cfg_init(&cfg)              // 创建编码配置对象
mpp_enc_cfg_set_s32(cfg, ...)       // 设置分辨率/帧率/码率/Profile 等
mpi->control(ctx, MPP_ENC_SET_CFG, cfg)  // 写入配置
    │                                  类比：ioctl 设参数
    ▼
mpi->control(ctx, MPP_ENC_GET_HDR_SYNC, &pkt)  // 获取 SPS/PPS
    │                                  这是 RTMP 推流必须的"解码说明书"


// ===== Buffer 管理 =====

mpp_buffer_group_get_internal(&grp, MPP_BUFFER_TYPE_NORMAL)
mpp_buffer_get(grp, &buf, frameSize)    // 分配编码器输入 buffer
    │                                    类比：V4L2 的 VIDIOC_REQBUFS + mmap
    ▼
mpp_frame_init(&frame)                  // 创建帧描述对象
mpp_frame_set_width(frame, 1280)
mpp_frame_set_height(frame, 720)
mpp_frame_set_fmt(frame, MPP_FMT_YUV420SP)  // NV12
mpp_frame_set_buffer(frame, buf)        // 关联 buffer
    │                                    类比：V4L2 的 buf.m.planes


// ===== 编码循环 =====

memcpy(mpp_buffer_get_ptr(buf), nv12Data, size)  // 写入 NV12 数据
    │                                  类比：mmap 后读 DMA buffer
    ▼
mpi->encode_put_frame(ctx, frame)    // 送一帧进编码器（异步）
    │                                  类比：V4L2 的 VIDIOC_QBUF
    ▼
mpi->encode_get_packet(ctx, &pkt)    // 取出编码后的 H.264 数据
    │                                  类比：V4L2 的 VIDIOC_DQBUF
    ▼
mpp_packet_get_pos(pkt)              // 获取 H.264 数据指针
mpp_packet_get_length(pkt)           // 获取数据长度
    │
    └─ 这就是可以直接推给 RTMP 的 H.264 NAL 单元数据


// ===== 清理 =====

mpp_frame_deinit(&frame)
mpp_buffer_put(buf)
mpp_buffer_group_put(grp)
mpp_enc_cfg_deinit(cfg)
mpp_destroy(ctx)                     // 类比：close(fd)
```

### 4.3 MPP 和 V4L2 的对比（帮助理解）

```
你已经很熟悉 V4L2 了，MPP 的思路几乎一样：

V4L2（采集）                          MPP（编码）
─────────────────────                 ─────────────────────
open("/dev/video11")                  mpp_create() + mpp_init()
ioctl(VIDIOC_S_FMT)                   mpp_enc_cfg + MPP_ENC_SET_CFG
ioctl(VIDIOC_REQBUFS) + mmap()        mpp_buffer_group + mpp_buffer_get()
ioctl(VIDIOC_QBUF)                    encode_put_frame()
ioctl(VIDIOC_DQBUF)                   encode_get_packet()
ioctl(VIDIOC_STREAMOFF) + close()     mpp_destroy()

核心区别：
  V4L2：数据方向是 硬件→用户（采集）
  MPP：数据方向是 用户→硬件→用户（编码）
```

### 4.4 编码参数说明

```
我们设置的关键参数：

  prep:width/height = 1280/720     输入分辨率
  prep:format = MPP_FMT_YUV420SP   NV12 格式（和 V4L2 采集输出一致）
  
  rc:mode = CBR                    恒定码率（Constant Bit Rate）
  rc:bps_target = 2000000          目标 2Mbps
  rc:bps_max = 2500000             最大 2.5Mbps（波动上限）
  rc:bps_min = 1500000             最小 1.5Mbps（波动下限）
  rc:gop = fps * 2                 GOP 长度 = 帧率×2 = 每 2 秒一个 I 帧
  
  h264:profile = 100               High Profile（压缩率最高）
  h264:level = 41                  Level 4.1（支持 1080p@30fps）
  h264:cabac_en = 1                CABAC 熵编码（比 CAVLC 压缩率高 10-15%）

CBR vs VBR：
  CBR：码率恒定，适合网络推流（带宽可预测）
  VBR：码率随画面复杂度变化，适合本地录像（省空间）
```

## 第五章：ffmpeg libavformat — RTMP 封装与推送

### 5.1 为什么用 ffmpeg

```
MPP 只负责编码：NV12 → H.264 裸流（NAL 单元）
但 RTMP 服务器不接受裸 H.264 流，需要封装成 FLV 容器格式

我们不手写 FLV 封装（格式太复杂），用 ffmpeg 的 libavformat：
  - 自动处理 FLV 封装
  - 自动处理 RTMP 握手和传输
  - 自动处理时间戳同步

实际只调了 5 个 ffmpeg API。
```

### 5.2 ffmpeg RTMP 推流 API 流程

```
// ===== 初始化 =====

avformat_alloc_output_context2(&fmtCtx, NULL, "flv", rtmpUrl)
  │  创建 FLV 输出上下文
  │  "flv" = 强制用 FLV 封装格式（RTMP 标准）
  ▼
avformat_new_stream(fmtCtx, NULL)
  │  添加一路视频流
  │  设置 codecpar：codec_id=H264, width, height, extradata=SPS/PPS
  ▼
avio_open(&fmtCtx->pb, rtmpUrl, AVIO_FLAG_WRITE)
  │  打开 RTMP 连接
  │  内部做了：DNS 解析 → TCP 连接 → RTMP 握手 → 创建流
  ▼
avformat_write_header(fmtCtx, NULL)
  │  写 FLV 文件头 + 发送 SPS/PPS


// ===== 推流循环 =====

av_init_packet(&avpkt)                  // 初始化 AVPacket
avpkt.data = mpp_packet_get_pos(pkt)    // 指向 MPP 输出的 H.264 数据
avpkt.size = mpp_packet_get_length(pkt) // 数据长度
avpkt.pts  = 帧序号 * 1000 / fps       // 时间戳（毫秒）
avpkt.flags = AV_PKT_FLAG_KEY           // 如果是 I 帧，标记关键帧
  │
  ▼
av_interleaved_write_frame(fmtCtx, &avpkt)
  │  将 H.264 数据封装成 FLV Tag → 通过 TCP 发给 RTMP 服务器
  │  内部自动做：FLV Tag 打包 + RTMP Chunk 分片 + TCP 发送


// ===== 清理 =====

av_write_trailer(fmtCtx)               // 写 FLV 尾部
avio_closep(&fmtCtx->pb)               // 关闭 RTMP 连接
avformat_free_context(fmtCtx)           // 释放上下文
```

### 5.3 时间戳为什么重要

```
RTMP 服务器和播放器靠时间戳（PTS）来：
  1. 决定每帧显示的时刻（不能太快也不能太慢）
  2. 计算缓冲进度（播放器需要知道"我该等多久"）
  3. 音视频同步（虽然我们只有视频）

我们的时间戳计算：
  pts = 帧序号 × 1000 ÷ fps
  第 0 帧 → 0ms
  第 1 帧 → 40ms（@25fps，1000/25=40）
  第 2 帧 → 80ms
  ...

  如果时间戳不对：
  - 递减或乱序 → 播放器卡顿/花屏
  - 间隔太大 → 播放器以为卡了，开始缓冲
  - 间隔太小 → 播放加速/跳帧
```

## 第六章：逐行解读 rtmp_streamer — 代码架构

### 6.1 类设计

```
RtmpStreamer 继承 QThread，运行在独立线程，不阻塞 UI：

CameraThread（采集线程）                RtmpStreamer（推流线程）
    │                                       │
    │ emit nv12FrameReady(ptr, size)        │
    │────────────────────────────▶ pushNv12Frame()
    │                                  ├─ memcpy 到 pendingFrame
    │                                  └─ wakeOne() 唤醒 run()
    │                                       │
    │                                  run() 循环：
    │                                  ├─ wait() 等新帧
    │                                  ├─ memcpy 到 MPP buffer
    │                                  ├─ encode_put_frame()
    │                                  ├─ encode_get_packet()
    │                                  └─ av_interleaved_write_frame()
```

### 6.2 线程间数据传递

```cpp
void RtmpStreamer::pushNv12Frame(void *nv12Data, int dataSize)
{
    if (!m_running || !m_streaming) return;
    
    QMutexLocker lock(&m_frameMutex);
    memcpy(m_pendingFrame, nv12Data, dataSize);   // 拷贝一份（采集线程的 buffer 马上要还给硬件）
    m_hasNewFrame = true;
    m_frameCond.wakeOne();                         // 唤醒推流线程
}
```

```
为什么要 memcpy 而不是直接传指针？

  CameraThread 的 V4L2 buffer 是 mmap 的——DQBUF 拿到数据后必须尽快 QBUF 还回去，
  否则硬件没 buffer 可写 → 丢帧。

  如果 RtmpStreamer 直接用指针，等它编码完（~2ms）的这段时间里，
  CameraThread 可能已经 QBUF 把 buffer 还回去了 → buffer 被硬件覆写 → 编码脏数据。

  所以必须 memcpy 到自己的 pendingFrame 再编码。
  720p NV12 = 1.38MB memcpy，ARM Cortex-A55 约 0.3ms，代价可忽略。
```

### 6.3 背压与错误处理

```
RTMP 推流可能失败的场景：
  1. 网络断开 → av_interleaved_write_frame 返回 < 0
  2. RTMP 服务器停了 → avio_open 失败
  3. 编码器异常 → encode_put_frame 返回非 MPP_OK

处理策略：
  - 初始化失败 → emit streamError() 通知 UI，线程退出
  - 推流中断 → break 退出循环，清理资源，线程结束
  - 没有自动重连（可以加，但增加复杂度）
```

## 第九章：面试 Q&A — RTMP 推流部分

### Q1：你的 RTMP 推流是怎么做的？

> 我用 Rockchip 的 **MPP 框架**做硬件 H.264 编码，再用 **ffmpeg 的 libavformat** 做 FLV 封装和 RTMP 推送。
> 摄像头采集线程每帧 emit 一个 NV12 原始数据信号，推流线程收到后通过 MPP 的 `encode_put_frame` 送入硬件编码器，
> 用 `encode_get_packet` 取出 H.264 NAL 数据，然后用 `av_interleaved_write_frame` 推到 RTMP 服务器。
> 整个编码过程在硬件 IP 核上完成，**CPU 占用几乎为零**。

### Q2：为什么不用 CPU 软编码（libx264）？

> 720p 用 libx264 在 ARM Cortex-A55 上大约需要 **30-50ms/帧**，单核 100% 占满，只能跑 20fps。
> 而 RK3588 的 MPP 硬件编码器 **1-2ms/帧**，CPU 占用趋近于零。
> 我们的嵌入式平台要同时跑雷达、温湿度、UI 渲染，不能让编码抢 CPU。

### Q3：MPP 的 API 流程和 V4L2 很像？

> 对，思路完全一样：`mpp_create` 对应 `open`，`mpp_enc_cfg` 对应 `VIDIOC_S_FMT`，
> `mpp_buffer_get` 对应 `VIDIOC_REQBUFS + mmap`，`encode_put_frame` 对应 `QBUF`，
> `encode_get_packet` 对应 `DQBUF`。都是**往硬件送数据、从硬件取结果**的生产者-消费者模式。
> 区别是 V4L2 的数据方向是硬件到用户（采集），MPP 是用户到硬件再到用户（编码）。

### Q4：SPS/PPS 是什么？为什么 RTMP 要特别处理？

> SPS 和 PPS 是 H.264 的**参数集**，告诉解码器分辨率、Profile、编码参数等元信息。
> 解码器没有 SPS/PPS 就无法解析 I 帧和 P 帧。
> RTMP/FLV 要求在第一个视频包之前就发送 SPS/PPS（放在 `extradata` 里），
> 我在 `initMppEncoder` 时用 `MPP_ENC_GET_HDR_SYNC` 从 MPP 获取，设到 `AVStream->codecpar->extradata`。

### Q5：推流线程和采集线程怎么协调的？

> 采集线程通过 `nv12FrameReady` 信号把 NV12 数据指针传过来。
> 推流线程的 `pushNv12Frame` 先 **memcpy 一份数据**到自己的 buffer（因为采集线程的 V4L2 mmap buffer 要尽快 QBUF 还回去），
> 然后通过 `QWaitCondition::wakeOne` 唤醒推流线程的 `run()` 循环去编码和推送。
> 这样采集线程不用等编码完成，编码线程有新帧才干活，没帧就 wait 休眠。

### Q6：如果网络断了怎么办？

> `av_interleaved_write_frame` 返回负值时，推流循环 break 退出，自动清理 MPP 编码器和 RTMP 连接。
> 目前没做自动重连——可以加一个重连逻辑（sleep 几秒后重新 init），但当前项目阶段先保持简单。

### Q7：MJPEG 和 RTMP 推流可以同时工作吗？

> 可以，互不干扰。MJPEG 消费的是 `frameReady(QImage)` 信号（RGB 数据），
> RTMP 消费的是 `nv12FrameReady(void*, int)` 信号（NV12 原始数据）。
> 两个信号在同一次 DQBUF 后同时 emit，分别走各自的处理链路。
> MJPEG 用 CPU 做 JPEG 编码（~5ms），RTMP 用 MPP 硬件做 H.264 编码（~0 CPU），不冲突。
